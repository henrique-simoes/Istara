# Istara Agent Entrypoint

Start here.

If you are an LLM or coding agent working on Istara, read this file first. Its job is to route you to the right documents in the right order so you can understand the system, preserve architecture, and keep the repo's own agents aligned with product changes.

## Compass

`Compass` is the name of Istara's comprehensive agentic development system.

When humans or agents say "follow Compass," they mean the combined system of:

- agent-entry guidance
- system prompts and governance rules
- generated architecture maps
- impact matrices
- change checklists
- technical narrative docs
- persona knowledge
- test and simulation maintenance expectations

Compass exists so Istara can scale development without losing architectural memory, UX consistency, CI/CD doctrine, release/install/update correctness, or agent comprehension.

## Mission

When making changes to Istara, you are responsible for all of the following:

1. implementing the code correctly
2. preserving backend/frontend/UX/system architecture
3. updating tests so future regressions are caught
4. updating the docs and prompts that future agents will rely on
5. updating Istara's own persona files when the product gains capabilities those agents need to understand
6. keeping release/version/update behavior coherent
7. keeping Compass itself current so the next agent inherits the new reality

If you only change code and do not update the surrounding system, the change is incomplete.

## Reading Order

## Non-Negotiable Outcomes

For meaningful changes, make sure:

- code is updated
- relevant tests are updated
- generated docs are regenerated
- `Tech.md` is updated when the system story changed
- persona files are updated when Istara's own agents need to understand the change
- release-worthy pushes to `main` publish the expected installer/release artifacts
- intentional release prep, tags, and manual dispatch remain internally consistent with that qualifying-main-push publishing flow

## Fast Decision Rules

When unsure, use these heuristics:

- If a backend payload changed, `frontend/src/lib/types.ts` probably changes too.
- If a user-visible workflow changed, a simulation scenario should probably change too.
- If a new workflow, installer path, onboarding state, or multi-step UX journey was introduced, a new simulation scenario may be required.
- If the system story changed, `Tech.md` should probably change too.
- If Compass would mislead the next agent after your change, update Compass in the same change.
- If Istara's own agents should talk about or use the feature, persona files should probably change too.
- If future agents will need to discover the new structure automatically, `scripts/update_agent_md.py` may need to change too.
- If you are preparing a release, use `./scripts/prepare-release.sh --bump` instead of hand-running a partial sequence.

## Minimum Completion Standard

For any meaningful product change, a good final check is:

1. Can the code execute correctly?
2. Can the UI and backend still agree on data and flow?
3. Can future tests catch regressions here?
4. Can future coding agents understand the change from the docs?
5. Can Istara's own agents understand and operate around the change?
6. Does Compass now describe the new reality clearly enough that the next agent will not guess?

If any answer is "no", the work is not done.
